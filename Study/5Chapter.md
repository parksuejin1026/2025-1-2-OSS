# Git 챕터 5: 버전, 태그 및 브랜치 관리 (PDF: 13, 14, 15)

## 1. 버전과 태그 (Tag) 활용

태그(Tag)는 프로그램의 **특정 커밋(해시 값)**에 **버전 번호**나 이름을 부여하는 기능입니다.

### 1.1. 소프트웨어 버전 방식 (SemVer)

| 항목 | 형식 | 설명 |
| :--- | :--- | :--- |
| **SemVer** | **`major.minor.patch`** | 세 자리 숫자 형태로 표기하는 버전 방식입니다. |
| **Major** | (첫째 자리) | 정식 버전은 1부터 시작하며, 하위 버전과 호환되지 않는 큰 기능 변경 시 올립니다. |
| **Minor** | (둘째 자리) | 메이저 버전에서 기능을 추가하거나 변경 사항이 있을 때 바꿉니다. |
| **Patch** | (셋째 자리) | 버그 수정 등 미미한 변화가 있을 때 바꿉니다. |

### 1.2. 태그 생성 및 관리 명령

| 명령 | 종류 및 설명 |
| :--- | :--- |
| **`$ git tag v1.0.1`** | **일반(가벼운) 태그**를 생성합니다. 태그 이름만 포함됩니다. |
| **`$ git tag -a v1.0.0 -m 'msg'`** | **주석 태그**를 생성합니다. 태그 이름, 작성자, 메시지 등의 정보가 포함됩니다. |
| **`$ git tag v1.1.0 HEAD~`** | **특정 커밋**에 태그를 붙입니다. (예: 현재 커밋의 이전 커밋) |
| **`$ git tag`** | 저장소의 **모든 태그 목록**을 표시합니다. |
| **`$ git show v1.0.0`** | 주석 태그의 **자세한 정보** (태그 작성자, 메시지, 커밋 내용)를 표시합니다. |
| **`$ git tag -d v1.0.0`** | 지정한 태그를 **삭제**합니다. |

## 2. 브랜치 (Branch) 개요 및 관리

브랜치는 하나의 프로젝트를 여러 갈래로 나누어서 **독립적으로 개발**을 진행할 수 있게 해주는 기능입니다.

### 2.1. 브랜치 기본 개념

* **기본 브랜치:** 저장소 생성 시 처음 만들어지는 브랜치이며, 현재는 보통 `main`을 사용합니다 (예전에는 `master`였습니다).
* **HEAD:** 현재 작업 중인 브랜치의 **최신 커밋**을 가리키는 포인터입니다.
* **브랜치 장점:** 저장소에서 다른 브랜치에 영향 없이 **새로운 기능 개발**이나 **버그 수정**을 안전하게 할 수 있습니다.
* **브랜치 병합 (Merge):** 독립된 브랜치에서 작업한 내용을 원래 버전과 합치는 과정입니다.

### 2.2. 브랜치 생성 및 이동 명령

| 명령 | 역할 및 설명 |
| :--- | :--- |
| **`$ git branch [bname]`** | 브랜치 **생성만** 하고 HEAD는 이동하지 않습니다. |
| **`$ git switch -c [bname]`** | 브랜치를 **생성하고 바로 새 브랜치로 이동**합니다 (권장). |
| **`$ git checkout -b [bname]`** | 브랜치 **생성하고 바로 이동**합니다 (이전 명령). |
| **`$ git switch [bname]`** | 지정한 브랜치로 HEAD를 이동합니다. |
| **`$ git checkout [bname]`** | 지정한 브랜치로 HEAD를 이동합니다 (이전 명령). |
| **`$ git switch -`** | **직전 브랜치**로 다시 이동합니다. |
| **`$ git checkout -`** | **직전 브랜치**로 다시 이동합니다 (이전 명령). |

### 2.3. 브랜치 목록 확인 및 삭제

| 명령 | 설명 |
| :--- | :--- |
| **`$ git branch`** | 현재 저장소의 **브랜치 목록**을 확인합니다 (`*`이 현재 작업 브랜치). |
| **`$ git branch -v`** | 브랜치마다 **마지막 커밋 ID와 메시지**도 함께 표시합니다. |
| **`$ git branch -d [bname]`** | **이미 병합된** 브랜치를 삭제합니다. |
| **`$ git branch -D [bname]`** | **병합되지 않은** 브랜치도 **강제로 삭제**합니다. |
| **`$ git branch --merged`** | 현재 브랜치 기준으로 **병합된** 브랜치 목록을 표시합니다. |
| **`$ git branch --no-merged`** | 현재 브랜치 기준으로 **아직 병합되지 않은** 브랜치 목록을 표시합니다. |

### 2.4. 분리된 HEAD (Detached HEAD)

* **개념:** HEAD가 현재 브랜치의 마지막 커밋이 아닌 **그 이전 커밋**이나 **특정 커밋 ID**를 가리키는 상태입니다.
* **이동 명령:** `$ git checkout HEAD~` 또는 `$ git switch -d HEAD~`.

## 👨‍💻 좋은 개발자가 되기 위한 Git 활용 심화 팁

### 1. 브랜치 전략 (Git Flow)의 이해

좋은 개발자는 브랜치 전략을 통해 프로젝트의 안정성을 유지합니다. Git Flow는 가장 유명한 전략 중 하나입니다.
* **`main` (또는 `master`):** **운영 환경에 배포된 안정된 코드**만 관리합니다. 태그(Tag)를 붙여 릴리즈 버전을 표시합니다.
* **`develop`:** 다음 배포 버전을 개발하기 위한 통합 브랜치입니다.
* **`feature/[기능명]`:** 새로운 기능 개발 시 `develop`에서 따서 사용하고, 개발 완료 후 `develop`에 병합합니다.
* **`hotfix/[버그명]`:** 운영 환경(main)의 긴급 버그 수정 시 `main`에서 따서 사용하고, 수정 완료 후 `main`과 `develop` 모두에 병합합니다.

### 2. 태그는 '릴리즈' 시점에 사용

태그는 단순히 커밋에 이름을 붙이는 것이 아니라, **"이 시점의 코드는 안정적이며 릴리즈할 준비가 되었다"**는 공식적인 표시입니다. 따라서 `main` 브랜치의 **주요 병합 커밋**에만 `vX.Y.Z` 형식으로 태그를 붙여야 합니다.

### 3. `git switch` 명령어 우선 사용

최신 Git에서는 **`git switch`**를 사용하여 **브랜치/커밋 이동**에 사용하고, **`git restore`**를 **파일 복원**에 사용하도록 역할을 분리하는 것이 좋습니다.
* `$ git switch -c [bname]` (새 브랜치 생성 및 이동)
* `$ git switch -d [이전커밋]` (과거 커밋으로 이동)
